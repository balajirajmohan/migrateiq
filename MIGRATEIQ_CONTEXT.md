# MigrateIQ — Cursor Project Context

## What Is This Project?

MigrateIQ is an **AI-agentic Oracle database migration platform** built for
banking and financial services enterprises. It uses a CrewAI multi-agent
architecture where 6 specialized AI agents plan, execute, monitor, validate,
and roll back Oracle DB migrations autonomously — with humans approving at 3
critical gates.

This is a POC (Proof of Concept) targeting enterprise clients who migrate
Oracle databases to new RAC clusters, with the primary pain point being
**migration downtime and risk**.

---

## Business Context

### The Problem
Banks run core banking systems (Temenos, Finacle, FIS) on Oracle. When
migrating to new hardware or RAC clusters:
- Downtime costs ~$95,000/minute
- Manual runbooks take 3 days per DB to write
- 23% of migrations require rollback
- Post-migration validation takes 8 hours manually
- Every data inconsistency is a regulatory incident

### The Solution
6 AI agents that replace the manual DBA workflow:
- Agents do the work
- Humans approve at 3 gates
- System learns from every migration

### Reference SOW (Real World Basis)
The SOW that inspired this POC:
- Client: DART (via mycompany reseller)
- Vendor: clientcompany Software & Services
- Scope: 37 Oracle DBs migrated to 4x RAC clusters
- 24 test DBs + 13 production DBs + 37 standby DBs
- Patch all 74 DBs from Oracle 19.10 → 19.24
- Fixed fee: $438,247
- Timeline: August 2025 – March 2026

## The 6 Agents — Detailed Spec

---

### Agent 1: Risk Assessment Agent

**File:** `agents/risk_agent.py`
**Role:** Pre-flight inspector — scans source DB, scores migration risk
**Triggered:** Start of migration
**Output:** `outputs/risk_report.md` + `metadata.json`

**Tools it uses:**
- `oracle_tools.py` → query DBA_* views
- `log_analyzer_tools.py` → Claude API analysis
- `report_tools.py` → format risk report

**Key Oracle queries it runs:**
```sql
-- Schemas
SELECT username, account_status FROM dba_users
WHERE account_status = 'OPEN'
AND username NOT IN ('SYS','SYSTEM','DBSNMP');

-- Table inventory
SELECT owner, table_name, num_rows, last_analyzed
FROM dba_tables WHERE owner NOT IN ('SYS','SYSTEM');

-- Object inventory
SELECT owner, object_type, COUNT(*) as cnt
FROM dba_objects
WHERE owner NOT IN ('SYS','SYSTEM')
GROUP BY owner, object_type;

-- DB Links (migration risk)
SELECT owner, db_link, host FROM dba_db_links;

-- Scheduled jobs (cutover conflict risk)
SELECT owner, job_name, next_run_date, enabled
FROM dba_scheduler_jobs WHERE enabled = 'TRUE';

-- Active sessions
SELECT username, status, COUNT(*) FROM v$session
WHERE username IS NOT NULL GROUP BY username, status;

-- Patch level
SELECT patch_id, patch_uid, version, action, status
FROM dba_registry_sqlpatch ORDER BY action_time DESC;

-- Storage per schema
SELECT owner, ROUND(SUM(bytes)/1024/1024/1024,2) AS size_gb
FROM dba_segments WHERE owner NOT IN ('SYS','SYSTEM')
GROUP BY owner ORDER BY size_gb DESC;
```

**Risk scoring logic (Claude API prompt):**
```
Given this Oracle database metadata: {metadata_json}
Migration path: Oracle 19.10 → 19.24, standalone → 2-node RAC

Score migration risk 0-100. Identify:
- BLOCKERS: must fix before migration can proceed
- WARNINGS: monitor during migration
- OK: no action needed

Return structured JSON:
{
  "risk_score": 0-100,
  "risk_level": "LOW|MEDIUM|HIGH|CRITICAL",
  "blockers": [{"issue": "", "recommendation": ""}],
  "warnings": [{"issue": "", "recommendation": ""}],
  "estimated_downtime_hours": 0,
  "summary": ""
}
```

### Agent 3: Execution Agent

**File:** `agents/execution_agent.py`
**Role:** The DBA who never panics — runs every step in the runbook
**Triggered:** After Human Gate #2 approval
**Communicates with:** Monitoring Agent via shared SQLite state

**Tools it uses:**
- `rman_tools.py` → backup/restore
- `datapump_tools.py` → expdp/impdp
- `dataguard_tools.py` → standby DB setup
- `ssh_tools.py` → remote execution
- `oracle_tools.py` → pre/post step checks

**Shared state schema (SQLite):**
```python
migration_state = {
    "migration_id": "uuid",
    "current_step": "step_id",
    "step_status": "pending|running|completed|failed",
    "health_score": 100,
    "active_alerts": [],
    "abort_requested": False,      # Monitoring Agent sets this True
    "steps_completed": [],
    "rman_backup_scn": None,       # Set after backup — rollback anchor
    "start_time": "ISO timestamp",
    "last_updated": "ISO timestamp"
}
```

**Abort check (runs before every step):**
```python
def check_abort_signal(migration_id):
    state = get_migration_state(migration_id)
    if state["abort_requested"]:
        raise MigrationAbortException("Abort requested by monitoring agent")
    if state["health_score"] < 60:
        raise MigrationAbortException(f"Health score critical: {state['health_score']}")
```

### Agent 4: Monitoring Agent

**File:** `agents/monitoring_agent.py`
**Role:** 2AM watchdog — continuously watches logs and DB health
**Triggered:** Same time as Execution Agent (parallel thread)
**Can trigger:** Rollback Agent automatically

**Tools it uses:**
- `watchdog` → tail Oracle alert log
- `Claude API` → classify log lines
- `oracle_tools.py` → poll V$ views
- `Slack SDK / smtplib` → send alerts

**Alert log analysis prompt:**
```
You are an Oracle DBA monitoring a live database migration.
Analyze these alert log lines and classify each:

Log lines:
{log_chunk}

For each line return:
{
  "timestamp": "",
  "severity": "INFO|WARNING|ERROR|CRITICAL",
  "error_code": "ORA-XXXXX or null",
  "description": "",
  "recommended_action": "",
  "trigger_rollback": true|false
}

Trigger rollback only for: ORA-00600, ORA-07445, ORA-04031,
data corruption indicators, or redo log failures.
```

**Health score calculation:**
```python
def calculate_health_score(metrics):
    score = 100
    if metrics["ora_errors_last_5min"] > 0:    score -= 20
    if metrics["ora_errors_last_5min"] > 5:    score -= 30
    if metrics["tablespace_usage_pct"] > 90:   score -= 15
    if metrics["redo_log_switches_per_hour"] > 20: score -= 10
    if metrics["active_blocking_sessions"] > 0: score -= 25
    if metrics["datapump_status"] == "ERROR":  score -= 50
    return max(0, score)
```

### Agent 5: Validation Agent

**File:** `agents/validation_agent.py`
**Role:** Post-migration auditor — proves nothing changed
**Triggered:** After Execution Agent signals completion
**Output:** `outputs/validation_report.xlsx`, `outputs/compliance_report.pdf`

**Validation checks:**

```python
validation_checks = [
    # Level 1 — Counts
    "row_count_match",          # every table source vs target
    "table_count_match",        # total tables per schema
    "object_count_match",       # all object types

    # Level 2 — Integrity
    "checksum_match",           # ORA_HASH per table
    "constraint_status",        # all ENABLED + VALIDATED
    "index_status",             # all VALID
    "invalid_objects",          # count = 0 after recompile

    # Level 3 — Sequences & Jobs
    "sequence_values",          # target >= source
    "scheduler_jobs_status",    # re-enabled correctly

    # Level 4 — Business Rules (Banking)
    "total_loan_portfolio",     # SUM(outstanding_amount)
    "active_account_count",     # COUNT of active customers
    "transaction_totals",       # SUM of daily transactions

    # Level 5 — Performance
    "top_query_response_time",  # within 10% of baseline
]
```

**Checksum query:**
```sql
-- Run on both source and target, results must match
SELECT
    '{table_name}' as table_name,
    COUNT(*) as row_count,
    SUM(ORA_HASH(
        {comma_separated_all_columns}
    )) as data_checksum
FROM {schema}.{table_name};
```

### Agent 6: Rollback Agent

**File:** `agents/rollback_agent.py`
**Role:** Safety net — auto or manual rollback to pre-migration state
**Triggered by:**
  - Monitoring Agent (health_score < 40 or trigger_rollback=True)
  - Human (abort button in UI)
  - Validation Agent (NO-GO decision)

**Tools it uses:**
- `rman_tools.py` → restore from backup SCN
- `oracle_tools.py` → reopen source DB
- `ssh_tools.py` → stop target processes
- `Claude API` → analyze failure + update knowledge base

**Rollback sequence:**
```python
rollback_steps = [
    "stop_datapump_jobs",
    "stop_dataguard_sync",
    "restore_source_from_rman_scn",    # uses rman_backup_scn from state
    "reopen_source_db",
    "re_enable_scheduler_jobs",
    "verify_source_healthy",
    "notify_team",
    "log_failure_to_knowledge_base",
    "generate_incident_report"
]
```

## Key Data Integrity Checks

```sql
-- 1. Row count baseline (run before AND after migration)
SELECT table_name, num_rows
FROM dba_tables
WHERE owner = :schema
ORDER BY table_name;

-- 2. Exact row count (accurate, use for critical tables)
SELECT COUNT(*) FROM :schema.:table;

-- 3. Checksum (data fingerprint — must match source vs target)
SELECT SUM(ORA_HASH(ROWID)) as checksum
FROM :schema.:table;

-- 4. Invalid objects (must be 0 after migration)
SELECT object_name, object_type, status
FROM dba_objects
WHERE owner = :schema AND status != 'VALID';

-- 5. Disabled constraints (must be 0 after migration)
SELECT constraint_name, constraint_type, status
FROM dba_constraints
WHERE owner = :schema AND status != 'ENABLED';

-- 6. Sequence sync check (target must be >= source)
SELECT sequence_name, last_number
FROM dba_sequences
WHERE sequence_owner = :schema;
```

## Key Design Decisions

1. **CrewAI sequential process** for phases 1-3 (risk → plan → execute)
2. **Parallel threads** for Execution + Monitoring agents
3. **SQLite shared state** — simple, no infra needed for POC
4. **Rollback scripts generated BEFORE execution** — not improvised
5. **RMAN SCN anchor** captured immediately before cutover — exact restore point
6. **Zero tolerance** for checksum mismatch — banking = no data loss
7. **Human gates are blocking** — agents wait indefinitely for approval
8. **Knowledge base** — every failure stored, future migrations learn from it

## Coding Guidelines for This Project

- Use `python-oracledb` thin mode by default (no Oracle client install needed)
- All SQL queries parametrized — never f-string user input into SQL
- Every tool function returns a string (CrewAI tool requirement)
- All agent outputs written to `outputs/` folder as files
- Shared state always read fresh from SQLite — never cached in memory
- Every subprocess call captures stdout + stderr — feed to monitoring agent
- Claude API calls always include system prompt defining the agent's role
- Use `async` for monitoring agent log tailing
- All secrets via `.env` + `python-dotenv` — never hardcoded
- Log every agent action to `migration_{id}.log` with timestamps
